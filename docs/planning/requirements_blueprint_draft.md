# ColorPhage 需求蓝图草案

## 1. 项目定位

### 1.1 一句话定义

ColorPhage 是一个面向科研论文配图的 R 配色包，通过少量参数为 `ggplot2` 等图形系统生成兼具高级感、区分度与发表适配性的配色方案。

### 1.2 核心价值

本包的核心不是单纯提供可用颜色，也不是简单复刻顶级期刊默认色板，而是提供一种带有审美判断的科研配色能力。

目标是让用户在不具备系统设计背景的情况下，也能快速得到：

- 足够好看
- 足够区分
- 符合科研审美
- 适合论文发表与学术汇报

### 1.3 与现有方案的差异

与常见配色包或配色网站相比，ColorPhage 的目标差异在于：

- 不只提供 palette 列表，而是提供“科研图形审美优化”
- 不只强调可区分或色盲友好，而是强调“高级感 + 科研感 + 区分度”
- 不只让用户自己挑选颜色，而是尽量用少量参数帮助用户自动得到高质量结果
- 不只模仿 Nature/Science 风格，而是做出比传统期刊默认配色更灵活、更有个性、但仍然像科研图的颜色系统

## 2. 用户问题与使用场景

### 2.1 用户当前痛点

目标用户通常面临以下问题：

- `ggplot2` 默认颜色过于普通，图看起来没有设计感
- 现有配色包虽然多，但高级感不足，或者选择成本过高
- 顶级期刊常见色板过于模板化、呆板，个性不足
- 用户希望图既专业、克制、可发表，又比常规科研图更漂亮
- 用户不会系统调色，希望通过很少的参数直接得到美观结果

### 2.2 典型场景

- 论文主图和补充材料配图
- 学术汇报、组会、答辩幻灯片
- 基金申请书中的统计图和示意图
- 多张图之间希望形成统一的视觉风格

## 3. 产品目标

### 3.1 总体目标

构建一个具有明确审美主张的科研配色系统，使用户能够以低学习成本获得具有统一风格和较高完成度的图形配色结果。

### 3.2 核心目标

- 提供高审美质量的科研图配色
- 提供足够稳定的区分度和图中层次
- 降低用户选择和试错成本
- 支持从快速调用到细粒度风格控制的逐级使用路径
- 优先兼容 `ggplot2`，后续扩展到其他绘图系统

### 3.3 非目标

以下内容暂不作为第一阶段核心目标：

- 一开始就兼容所有 R 绘图生态
- 一开始就做完全自由的 AI 配色生成
- 一开始就覆盖海量 palette 数量
- 将产品定位为纯色盲检查工具或纯期刊复刻工具

## 4. 产品结构

当前讨论结果建议将包设计为三个层级：

1. 简单模式
2. 进阶模式
3. 定制生成模式

这样可以兼顾新手用户、熟练用户和有明确审美诉求的用户。

## 5. 简单模式

### 5.1 定位

简单模式的目标是让用户只回答一个非常直观的问题：

“我想要低调高级，还是想要醒目有冲击力？”

### 5.2 家族划分

简单模式当前定为只包含两种主模式：

#### A. `soft` 模式

中文解释名：淡雅

特征：

- 色彩淡雅
- 颜色彼此接近、和谐
- 不要求同一色系
- 整体克制、柔和、精致
- 适合多数论文和正式科研图

规则边界：

- 整体以低到中等饱和度为主
- 相邻颜色之间保留差异，但不过分跳跃
- 允许冷暖混合，但整体语气必须统一
- 默认避免刺眼的高亮色和过强冲突
- 强调和谐、精致、耐看，而不是单个颜色的存在感

设计目标：

- 适合大多数正式论文图和需要长期观看的科研图
- 让图显得经过设计，但不过于张扬

#### B. `vivid` 模式

中文解释名：强烈

特征：

- 色彩反差明显
- 对比强
- 不限制统一色系
- 需要有视觉冲击但不能显得廉价
- 适合需要强调组别差异或更强展示效果的场景

规则边界：

- 至少包含 1 到 2 个明确的视觉锚点
- 允许更大的色相跨度
- 明度与饱和度对比更明显
- 要保持整体秩序，避免杂乱、廉价或“彩色糖豆感”
- 强调醒目、鲜明、有表现力，但仍需符合科研图语境

设计目标：

- 适合需要强调组别差异、结果对比和展示效果的场景
- 在更强的视觉表现力下仍保持科研审美边界

### 5.3 简单模式使用体验

简单模式应尽量做到：

- 参数极少
- 默认结果稳定
- 用户无需理解复杂配色理论
- 给定 `n` 和模式后即可得到高质量颜色组合
- 用户只需在“淡雅”和“强烈”两种视觉策略之间做选择

### 5.4 简单模式参数建议

当前建议简单模式的对外接口优先保持极简：

```r
phage_palette(n = 5, mode = "soft")
phage_palette(n = 5, mode = "vivid")
```

设计原则：

- 第一版不暴露过多控制项
- 第一版优先提供稳定默认值
- 简单模式内部可以使用多套种子 palette 或生成规则
- 内部可以存在若干隐含子风格，但暂不直接暴露给用户

### 5.5 简单模式实现思路

当前建议简单模式不是简单绑定两套固定色板，而是：

- 两种主模式对应两种核心视觉策略
- 每种模式下可存在多个内置种子 palette
- 每种模式也可基于固定规则在有限范围内生成变体

这样可以兼顾：

- 使用体验上的简单
- 输出结果上的丰富性
- 后续扩展时的可维护性

### 5.6 简单模式待敲定问题

- 两类模式是否各自包含少量内部变体
- 是否允许用户额外输入主色进行引导
- 简单模式是否允许可选参数如 `accent` 或 `seed_family`

## 6. 进阶模式

### 6.1 定位

进阶模式基于 palette family 设计。用户不只选择“淡雅/强烈”，而是选择具体风格家族，从而获得更明确的视觉人格。

### 6.2 目标

- 建立一组具有明确审美差异的风格家族
- 允许用户在统一科研语境下选择不同气质
- 保证每个 family 都能稳定扩展到多个颜色数量场景

### 6.3 第一批 family 定稿

第一批 MVP family 暂定为以下 6 个：

- `mist`
- `mineral`
- `forest`
- `ocean`
- `cinema`
- `retro`

它们共同构成第一版进阶模式的核心风格系统，要求彼此差异明确，同时都保留“科研图语境下的高级感”。

### 6.4 family 定义与科研场景示例

#### A. `mist`

风格定义：

- 雾感、柔和、低饱和、细腻
- 适合克制、安静、长期观看的科研图
- 是简单模式 `soft` 的重要延伸

视觉边界：

- 饱和度倾向：低到中低
- 明度倾向：中高
- 冷暖倾向：中性偏冷或轻微偏暖，避免极端冷暖对撞
- 对比强度：低到中等
- 整体印象：柔和、轻盈、安静、精致

适用科研实例场景：

- 一张 RNA-seq 差异表达结果的分组柱状图，需要展示对照组与多个处理组之间的表达变化，但希望图整体温和、精致，不显得刺眼

最适合的图类型：

- 分组柱状图
- 分组折线图
- 分类散点图
- 配色需要长期观看的主文图

不适合的场景：

- 需要极强组间冲突的图
- 类别数很多且必须迅速强区分的图
- 需要强展示冲击力的封面感 figure

用户记忆锚点：

- 像清晨薄雾中的颜色，彼此和谐但并不单调

#### B. `mineral`

风格定义：

- 冷静、理性、克制、矿物感、偏蓝灰
- 强调现代、硬质、结构化的科研气质
- 适合方法学、统计分析和机制类图形

视觉边界：

- 饱和度倾向：低到中等
- 明度倾向：中等
- 冷暖倾向：明显偏冷
- 对比强度：中等
- 整体印象：硬质、现代、理性、可信赖

适用科研实例场景：

- 一张多组临床指标比较的箱线图，需要在保持专业和理性气质的同时，让不同组别清楚可分，又不显花哨

最适合的图类型：

- 箱线图
- 小提琴图
- 机制示意配套统计图
- 方法比较图

不适合的场景：

- 需要明显温暖叙事感的图
- 希望传达强有机生命力的生态/菌群主题图
- 需要非常活泼或亲和的展示型图

用户记忆锚点：

- 像实验台上的矿石与金属，冷静、稳定、有结构感

#### C. `forest`

风格定义：

- 自然、有机、生物感
- 带有植物、菌群、生态系统的联想
- 适合生命科学相关主题，但要避免变成“环保宣传风”

视觉边界：

- 饱和度倾向：中低到中等
- 明度倾向：中等
- 冷暖倾向：中性偏暖或自然中性
- 对比强度：中等
- 整体印象：自然、有机、稳健、有生命力

适用科研实例场景：

- 一张肠道菌群组成的堆叠柱状图或群落比较图，需要让不同菌群分类既有自然生物气息，又保持论文图的整洁和克制

最适合的图类型：

- 微生物群落比较图
- 堆叠柱状图
- 生态组成图
- 生物分组散点图

不适合的场景：

- 纯方法学或工程化气质很强的图
- 需要冷感、金属感的统计图
- 需要极高戏剧反差的结果展示图

用户记忆锚点：

- 像林地和苔藓的层次，有生命力但不喧闹

#### D. `ocean`

风格定义：

- 清爽、流动、蓝绿层次明显
- 比 `mineral` 更有生命力，比 `forest` 更冷静洁净
- 很适合做连续和渐变方向的 family

视觉边界：

- 饱和度倾向：中低到中等
- 明度倾向：中高
- 冷暖倾向：偏冷
- 对比强度：中低到中等
- 整体印象：清洁、流动、透明、有层次

适用科研实例场景：

- 一张代谢物丰度热图或环境因子梯度图，需要用连续颜色体现变化层次，同时保持清洁、流动和可信赖的科研视觉效果

最适合的图类型：

- 热图
- 连续变量梯度图
- 气泡图
- 环境因子或浓度变化图

不适合的场景：

- 需要温暖、怀旧叙事感的图
- 需要非常强烈离散对比的多组分类图
- 需要厚重深色主导的视觉风格

用户记忆锚点：

- 像海水从浅滩到深海的变化，清晰、流畅、有层次

#### E. `cinema`

风格定义：

- 有戏剧张力、深浅碰撞、强调视觉叙事
- 是简单模式 `vivid` 的精致化延伸
- 适合需要突出重点或呈现结果冲击力的图

视觉边界：

- 饱和度倾向：中等到中高，但需受控
- 明度倾向：中低到中高并存，允许拉开层次
- 冷暖倾向：可跨冷暖，但必须有主导语气
- 对比强度：中高
- 整体印象：聚焦、戏剧化、鲜明、有叙事感

适用科研实例场景：

- 一张药物处理前后肿瘤体积变化的折线图或分组散点图，需要让处理组和对照组之间的差异在视觉上更有冲击力，同时又不能像商业海报

最适合的图类型：

- 对照与处理组对比图
- 关键结果折线图
- 分组散点图
- 需要突出主结论的主图

不适合的场景：

- 需要非常克制、中性、背景化的图
- 类别非常多且每组都要均匀表达的图
- 需要极其温和的补充材料图

用户记忆锚点：

- 像电影打光，主体明确，情绪更强，但整体仍有秩序

#### F. `retro`

风格定义：

- 温暖、复古、胶片感、旧纸张感
- 提供一条不同于冷色科研图的审美路径
- 适合更具叙事性或展示感的图形

视觉边界：

- 饱和度倾向：中低到中等
- 明度倾向：中等偏暖
- 冷暖倾向：偏暖
- 对比强度：中低到中等
- 整体印象：温暖、沉静、有人情味、有时间感

适用科研实例场景：

- 一张历史样本来源分布图、物种比较图或综述型示意统计图，希望整体更温和、有故事感，但仍然保持论文图的整洁和可信度

最适合的图类型：

- 综述型统计图
- 样本来源或历史分布图
- 物种比较图
- 用于汇报或综述文章的配图

不适合的场景：

- 强调现代冷感的理工类方法图
- 需要极清爽冷色渐变的热图
- 需要明显高冲击对比的主结果图

用户记忆锚点：

- 像旧胶片和书页里的颜色，温暖、柔和、有时间感

### 6.5 第二梯队扩展候选

以下方向可作为后续扩展 family 候选，但暂不纳入第一版 MVP：

- `ink`：深色、墨感、克制、高反差，适合少组别高完成度图
- `neonlab`：更鲜明、更前卫、更适合展示和封面图，但科研边界更难控制

### 6.6 进阶模式待敲定问题

- 每个 family 的定义和边界是否还需进一步收紧
- family 之间如何避免重叠
- family 命名风格是否走画面感路线
- 是否为每个 family 同时提供离散、连续、发散版本

## 7. 定制生成模式

### 7.1 定位

定制生成模式用于满足更具体的审美需求。它不是完全无边界的自由生成，而是“受控生成”。

### 7.2 当前建议拆分

可拆分为两个方向：

#### A. 内置风格直接调用

- 按 family 调用固定 palette
- 按风格标签调用预设模板
- 保证结果稳定、可复现、易维护

#### B. 基于用户指令生成

- 用户输入主色
- 用户输入风格关键词
- 用户指定偏冷、偏暖、偏柔和、偏戏剧化等倾向
- 系统基于内部规则生成一个符合要求的 palette

### 7.3 第一阶段原则

第一阶段应优先做“受控生成”而不是完全开放生成：

- 先让输出稳定
- 先建立可解释的规则体系
- 先保证科研审美一致性
- 避免生成结果不可预测或风格失控

### 7.4 定制模式待敲定问题

- 第一期是否开放自然语言 prompt
- 是否只支持主色扩展
- 如何约束生成结果保持“科研感”
- 如何定义生成模式的最小参数集合

## 8. 审美原则与内部设计规范

这一部分将作为产品灵魂，后续需要单独整理为内部设计规范文档。

### 8.1 核心审美原则

- 不追求颜色数量多，而追求组合完整、协调、有秩序
- 不追求品牌视觉或互联网 UI 风，而要保留科研图的理性气质
- 默认偏低到中等饱和度，但必须有视觉锚点
- 高对比不等于高刺眼
- 柔和不等于发灰、发脏、没精神
- 颜色必须服务于图形结构，而不是抢夺注意力
- 好 palette 的标准不是单个颜色漂亮，而是整组放在一张 figure 中依然成立

### 8.2 应避免的方向

- 彩虹色连续渐变
- 多个高纯度亮色同时竞争视觉焦点
- 商业宣传感过强的撞色
- 明显不适合论文版面的“社交媒体风”视觉效果
- 复刻期刊默认色板到缺乏个性

### 8.3 科研适配原则

- 在论文图中应显得自然、不跳戏
- 需要在缩小显示后仍保持分组可辨识
- 在投影、屏幕和打印条件下尽量稳定
- 图例、线条、填充、点和误差线之间要有合理搭配

### 8.4 后续建议补充的量化指标

- 最小颜色距离阈值
- 默认明度范围
- 饱和度控制区间
- 强调色数量上限
- 连续色板层次均匀性

## 9. 命名与品牌方向

### 9.1 当前定稿方向

命名体系采用“三层分离”方案：

1. 包名：稳定、可长期使用、保留一定专业感
2. family 名：有画面感、能清楚传达风格气质
3. palette 名：最自由，承担轻松幽默和文化联想

这一结构用于平衡：

- 产品系统性的稳定
- 品牌识别度
- 文档和展示中的趣味性
- 后续公开发布与长期维护的可行性

### 9.2 命名目标

- 耳熟能详
- 容易记忆
- 与色彩风格有明确联想
- 让包更有人格和传播性
- 同时保留工程上的稳定性和可维护性

### 9.3 包名方案

当前建议继续使用：

- `ColorPhage`

保留原因：

- 名字独特，有辨识度
- 带有科研感，不会过于消费级或工具化
- 与项目“为科研图构建颜色系统”的气质相符
- 虽然不是一眼即懂功能，但可通过副标题和文档补足解释

### 9.4 family 命名策略

family 命名采用画面感词汇，而不是直接使用 IP 名或角色名。

当前第一批 family 命名保持为：

- `mist`
- `mineral`
- `forest`
- `ocean`
- `cinema`
- `retro`

命名原则：

- 单个英文词
- 有明确画面感
- 能直接传达风格差异
- 不依赖具体文化背景才能理解
- 适合作为长期稳定的系统分类名称

### 9.5 palette 命名策略

palette 名承担趣味性和文化联想，但不直接作为底层唯一标识。

当前建议采用双层结构：

- 内部稳定 ID：中性、规范、便于维护
- 对外展示名：轻松幽默、有文化联想或影射意味

例如：

- 内部 ID：`mist_01`
- 展示名：`Rainy Bus Stop`

或：

- 内部 ID：`cinema_02`
- 展示名：`Electric Hero`

### 9.6 palette 命名风格建议

palette 展示名可以采用以下方向：

- 经典电影、经典游戏、经典歌曲、经典角色的联想式命名
- 尽量偏“影射式”或“文化联想式”而不是强依赖商标名称
- 名字要与颜色气质有明显关联

建议优先采用：

- 原创展示名 + 文化灵感说明

而不是：

- 完全依赖直接 IP 名作为唯一正式命名

这样可以兼顾：

- 记忆点
- 趣味性
- 风格辨识度
- 更低的公开发布风险

### 9.7 family 与文化联想来源的关系

不同 family 可以有各自更偏好的文化联想来源，用于帮助 palette 展示名保持统一气质，但不应做成硬绑定规则。

原则：

- 每个 family 可以有主要的命名灵感池
- 命名灵感池用于增强系统一致性和记忆点
- 最终命名仍以颜色风格匹配度为最高优先级
- 如果某个展示名更适合另一个 family，可以允许跨来源使用

当前建议的命名灵感倾向：

- `mist`：温柔、安静、治愈感的电影、动画、日常场景联想
- `mineral`：科幻、宇宙、金属、机械、未来感作品联想
- `forest`：自然、奇幻、冒险、生命力相关的作品或场景联想
- `ocean`：海洋、梦境、水域、天空、流动感相关联想
- `cinema`：戏剧性强、角色张力强、光影感强的电影化联想
- `retro`：老电影、老歌、胶片、怀旧文化符号联想

### 9.8 当前待讨论问题

- 每个 family 下 palette 展示名的具体风格和数量
- palette 展示名是否需要统一灵感来源体系
- 展示名是否在函数层直接暴露，还是主要出现在文档和 gallery 中

## 10. API 初步设想

以下为概念级 API 草案，后续需要进一步收敛。

### 10.1 第一版 API 总体策略

第一版 API 建议采用“一个主入口 + 少量适配函数”的结构，避免用户在一开始面对多个并列函数造成认知负担。

当前建议保留：

- `phage_palette()`：主入口，返回颜色向量或 palette 对象
- `scale_color_phage()`：适配 `ggplot2` 的颜色映射
- `scale_fill_phage()`：适配 `ggplot2` 的填充映射
- `theme_phage()`：主题函数，作为后续增强接口

设计理由：

- 用户只需记住一个核心函数
- 简单模式、进阶模式和定制生成模式可以统一放在同一接口下
- 包整体会更像一个完整系统，而不是多个松散工具函数

### 10.2 主入口函数草案

```r
phage_palette(
  n,
  mode = NULL,
  family = NULL,
  palette = NULL,
  base = NULL,
  vibe = NULL
)
```

参数说明草案：

- `n`：所需颜色数量
- `mode`：简单模式入口，当前为 `soft` 或 `vivid`
- `family`：进阶模式入口，如 `mist`、`mineral`
- `palette`：具体 palette 的内部稳定 ID，高级或内部使用优先
- `base`：用户指定主色，用于受控生成
- `vibe`：用户指定生成倾向，如偏柔和、偏戏剧化、偏冷等

### 10.3 模式映射关系

主入口内的三条主要路径：

- 简单模式：`mode = "soft"` 或 `mode = "vivid"`
- 进阶模式：`family = "mist"` 等 family 参数
- 定制生成模式：由 `base`、`vibe` 等参数触发

原则上：

- 新手优先使用 `mode`
- 熟练用户优先使用 `family`
- 有明确审美需求的用户才使用 `base` 或 `vibe`

### 10.4 返回颜色向量

```r
phage_palette(n = 5, mode = "soft")
phage_palette(n = 6, mode = "vivid")
phage_palette(n = 5, family = "moonlight")
phage_palette(n = 6, base = "#3E6FB6", vibe = "soft")
```

说明：

- 第一版示例中的 family 名后续应与定稿 family 保持一致
- `palette` 参数可在后续展示命名系统定稿后补入更具体示例

### 10.5 应用于 ggplot2

```r
p + scale_color_phage(mode = "soft")
p + scale_fill_phage(family = "mist")
```

### 10.6 可能的主题支持

```r
p + theme_phage()
```

### 10.7 第一版建议公开参数范围

第一版建议优先公开和重点文档化的参数：

- `n`
- `mode`
- `family`

第一版可保留但不必作为重点宣传入口的参数：

- `palette`
- `base`
- `vibe`

设计考虑：

- 先保证最常用路径足够简单
- 先让大多数用户通过 `mode` 和 `family` 就能完成任务
- 受控生成能力可以先做成轻量高级功能，避免用户一开始误解为“任意 prompt 调色器”

### 10.8 API 设计原则

- 参数少
- 默认值强
- 名称直观
- 模式分层明确
- 同一风格在不同函数中尽量保持一致
- 避免第一版暴露过多生成控制项
- 尽量让用户不需要理解内部配色机制也能直接使用

### 10.9 API 待敲定问题

- `palette` 是否在第一版就直接对用户公开
- `base` 和 `vibe` 的命名是否足够直观
- 定制生成模式是否在第一版就开放
- `theme_phage()` 是否与第一版同步推出
- `ggplot2` 接口是否只先支持 `scale_color_*` / `scale_fill_*`

## 11. 技术范围与开发优先级

### 11.1 第一阶段建议范围

第一阶段优先实现：

- 明确包定位与审美规范
- 建立简单模式
- 建立第一批 family
- 提供高质量离散 palette
- 完整支持 `ggplot2`
- 建立最小可用 API
- 准备 gallery 或 vignette 展示效果

### 11.2 第一阶段不宜过度扩张的内容

- 过早支持太多绘图系统
- 过早开放完全自由的生成机制
- 过早追求 palette 数量规模
- 过早做复杂 UI 或网页配色工具

### 11.3 技术实现建议

优先将系统拆为以下内部模块：

- `easy palettes`
- `family palettes`
- `generator`
- `validator`
- `ggplot adapters`

## 12. 第一版 MVP 建议

当前建议的第一版 MVP：

- 简单模式：2 个主模式
- 进阶模式：6 到 10 个 family
- 离散 palette 为优先
- 连续和发散 palette 先做少量高质量版本
- `ggplot2` 完整支持
- 提供统一的 gallery 展示

### 12.1 MVP 成功标准

如果第一版满足以下标准，可视为方向正确：

- 用户能用极少参数得到“明显比默认 ggplot 更高级”的图
- 简单模式具备足够清晰的视觉差异
- 不同 family 之间风格差异明确
- 图在常见科研场景中可直接使用
- 包具有可延展性，后续能继续增加 family 与生成能力

## 13. 当前需要逐条敲定的议题

建议后续按如下顺序逐项讨论：

1. 包名是否定稿为 `ColorPhage`
2. 简单模式最终命名与规则边界
3. 进阶模式第一批 family 列表
4. family 的命名风格与 palette 命名风格
5. 审美规范文档的详细条目
6. API 的主函数结构
7. MVP 的具体开发流程与里程碑

## 14. 当前结论摘要

截至本草案版本，已基本达成的方向包括：

- 产品核心是“好看并且足够区分，并且符合科研审美可发表”
- 应先建立内置的 palette family，而不是一开始追求完全自由生成
- 产品应分为简单模式、进阶模式和定制生成模式
- 简单模式目前优先围绕“淡雅”和“强烈”两类展开
- 进阶模式要围绕风格家族构建系统
- 定制生成模式应采取受控生成策略
- 审美原则必须写成内部设计规范
- 命名系统可以更轻松、有幽默感和文化联想，但需注意结构与维护性
